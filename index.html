LAN Multiplayer over the Internet

Play LAN-multiplayer Windows games with a friend across the internet — one hosts, one joins with a code. Updated 2026-07-14.

What it is

A feature that lets two Bannerlator users play a LAN-multiplayer game together over the internet. One player hosts and gets a short 6-character code; the other joins with it. Both then launch the same game and use its normal LAN / Local Multiplayer menu — and find each other as if they were on the same home router.

It is a network layer, not a multiplayer layer: the game supplies the multiplayer; we make "LAN" reach across the internet. It works for any game with a LAN / direct-IP mode, and adds nothing to the Wine container — no Mono, no Gecko, no winetricks. If a game runs solo in the container, it runs the same with the overlay on.

How it works

Both phones are placed on one virtual LAN (10.99.0.0/24 — host .1, client .2). The game's LAN discovery broadcasts and direct connections are captured by a phone-level VPN, tunnelled through a relay to the other player, and re-injected on their side. The game never knows the internet is in between.

Game (inside Wine) "Host / Join LAN game" │ broadcast + direct-connect ▼ Wine winsock ─► Android app-UID socket ▼ Per-app VpnService tun (scoped to Bannerlator only) ▼ Native tunnel (liblannet.so): classify + forward │ ▼ internet Relay VPS ───────────────► friend's phone ─► their game a dedicated relay server (re-injected as local LAN)

Design choice: we built our own minimal 2-peer overlay rather than embedding a mesh (ZeroTier/n2n) — no licensing entanglement, full control, and it reuses our existing Cloudflare Worker + account infrastructure.

The pieces

📱 Overlay (VpnService)

A per-app Android VPN, scoped to Bannerlator only, that captures the game's LAN traffic. Device-proven.

⚙️ Native tunnel (liblannet.so)

C library (JNI) that reads the VPN, classifies each packet (broadcast → fan-out, direct → forward), and pumps it to/from the relay.

🛰️ Relay server

A tiny UDP forwarder on a free Oracle Cloud VM. Both players connect outbound to it, so it works through any home/mobile NAT. Live now.

🔑 Room-code signaling

Cloudflare Worker endpoints (/lan/host, /lan/join) that mint the 6-char codes and hand out the relay address. Deployed & live.

🎛️ App UI

Host / Join menus in the app — being polished into a native Compose UI synced across three entry points.

🔗 Shared session state

One live session, three views onto it (StateFlow). Start hosting anywhere → every menu shows it. Stop anywhere → all update.

🎮 Steam co-op auto-config

For Steam co-op games whose multiplayer uses Steam networking: a one-tap toggle auto-points their peer discovery at the overlay, so co-op finds a friend over the internet with zero manual setup. Toggle live & device-proven; end-to-end play pending a second device.

Proven end-to-end on real hardware: the polished in-app menu — Host a game → a real 6-character code (with Copy / Share) → overlay up → the live relay logs the phone's connection. All three entry points (nav drawer, per-game long-press, in-game) stay in sync via one shared session — hosting from any surface flips the others to "active". Separately, a broadcast was proven to cross between two peers through the relay over the real internet. Every mechanism the feature relies on now works on-device.

What's left

  • Polished UI — done & device-verified: a native Compose menu matching the app's design system, with a big legible/copyable room code, in the nav drawer, per-game long-press menu, and the in-game side menu — all reading from one shared session so they never drift. Plus a tappable status pill in the header that goes connecting → waiting → connected when a second player actually joins, and an optional "Hosted by <name>" for signed-in hosts (anonymous otherwise). All earlier UI follow-ups are now done.
  • Reliability / scaling — hardened: the relay now auto-frees rooms after a few minutes of silence (so dead rooms don't pile up) and its room capacity was raised. Comfortable for beta/community scale; clear cheap paths to larger scale documented.
  • Two-device proof (P1.6) — the finish line: install a known LAN game on two devices, host on one, join on the other, confirm they find each other and play. This is the true finish line — it needs a second device.
  • Encryption: a Noise handshake is added before any public release so the relay only ever sees ciphertext — a required item on the pre-launch checklist.
  • IPv6 & per-game quirks: currently IPv4-only; some games (e.g. Source-engine titles) host on IPv6 and would need IPv6 support. A few titles with hardcoded LAN subnets/ports may need small per-game tweaks.

What works, what doesn't

✓ Works

Any game with a LAN / Local / Direct-IP mode — the classic LAN-party catalogue (Age of Empires, Warcraft III, older CoD/Halo PC, Stardew/Terraria co-op, Minecraft "Open to LAN", etc.). Both broadcast-discovery and type-in-an-IP styles.

✗ Doesn't

Single-player-only games; online-only games that need official matchmaking servers and have no LAN mode; titles whose anti-cheat blocks VPN/tunnel interfaces.

Rule of thumb: if two people on the same physical router could play it, this lets them play it over the internet.
